# 给人类开发者的导读 (Getting Started for Humans)

欢迎来到全新的 **微内核/微前端游戏平台** 项目。

## 1. 为什么我们要重构？

旧版项目（目前存放在 `_old` 目录下）虽然实现了丰富的功能（GuessFunc 猜函数、关卡编辑器、模组系统等），但在架构上存在严重的“网状耦合”：
- **全局状态滥用**：UI 组件深度绑定了全局状态（如 `useGameStore`），导致组件无法单独测试或复用。
- **逻辑与视图混杂**：像 `GraphRenderer` 这种本应是纯粹渲染图表的组件，内部包含了大量的业务判题逻辑。
- **AI 与开发者极不友好**：修一个 Bug 往往需要理解整个系统，牵一发而动全身。

为了解决这些问题，我们引入了**微内核架构（Microkernel Architecture）**。

## 2. 我们的核心目标

本项目的目标是**100% 还原旧版的所有功能**，但在底层架构上进行脱胎换骨的重塑：
1. **极致解耦**：将系统划分为“必须有的基础内核”与“增量更新的游戏模组”。
2. **AI 友好**：确保 AI（或新加入的开发者）在修改某个模组时，**只需要阅读该模组的代码**，而不需要了解全局上下文。
3. **消除屎山**：我们绝不照抄旧版的网状代码。旧版中写得好的独立逻辑（如 `mathEngine` 纯数学解析核心）可以直接迁移并适配；但涉及 UI 和状态绑定的代码，必须按照新规范重构。

## 3. 如何阅读这份文档？

- **`rules/` 目录**：在写下任何一行代码前，请务必阅读这里的架构底线和编码规范。**这里定义了哪些行为是绝对禁止的**（如硬编码、临时补丁、跨模组 import）。
- **`proj/` 目录**：这里定义了系统的整体架构蓝图（内核、基础模组、共享模组、游戏模组及其子模组）。
- **`tasks/` 目录**：这里是我们将旧版功能拆解后的一系列迁移与重构任务。它是动态的，会随着项目的推进实时更新。

## 4. 实时更新原则

这四个文件夹（`for_human`, `rules`, `proj`, `tasks`）是项目的**单一事实来源 (Single Source of Truth)**。
任何架构变更、新增模组或任务状态改变，都必须**第一时间**反映在这些 Markdown 文档中。代码只是文档的实现。
